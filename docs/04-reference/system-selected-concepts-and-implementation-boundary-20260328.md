# 系统选用概念与实现边界总图 / 2026-03-28

## 1. 这份文档解决什么问题

这份文档不再顺着零散术语逐条打勾，而是把当前系统真正承担的概念，按四层重新整理：

1. **市场本体层**
2. **触发与结构层**
3. **执行与风险层**
4. **交易业务层**

它只回答一个问题：

`截至 2026-03-28，哪些概念已经被当前系统正式定义并承担，哪些只是部分吸收，哪些还只是书义、旧系统经验或待验证假说。`

## 2. 判定规则

### 2.1 已正式定义并实现

同时满足以下三点：

1. 代码里有明确合同、字段、枚举、trigger 或 runner。
2. 现行 `design / spec / conclusion` 已冻结。
3. 可以通过当前仓库代码与正式执行文档复核。

### 2.2 已部分承担，但不是完整实现

出现以下任一情况：

1. 只有局部 trigger 或局部 runner 在使用。
2. 有相近表达，但还没有统一系统合同。
3. 只吸收了书里概念的一部分，不等于整套原义已经落地。

### 2.3 当前未正式实现

出现以下任一情况：

1. 只有书义来源，没有新系统正式合同。
2. 只有旧系统实验，没有当前主线继承。
3. 目前仍属于研究假说、待验证命题或未来治理题目。

## 3. 先给总判断

### 3.1 当前系统真正已经承担的核心

1. `MALF` 三层主轴：
   - `monthly_state_8`
   - `weekly_flow_relation_to_monthly`
   - `pas_trigger`
2. A 股日线频率下的多重时间框架分析：
   - 月线背景
   - 周线顺逆
   - 日线触发
3. 五个 `PAS` trigger 的代码定义：
   - `BOF / BPB / PB / TST / CPB`
4. 主线做多执行合同：
   - `T+1` 开盘
   - `1R`
   - 固定名义仓
   - 整手约束
   - 部分止盈
   - trailing stop
5. `16` 格验证框架：
   - 用统一市场背景与顺逆关系，对 trigger 做正式 readout

### 3.2 当前系统没有承担整套“交易哲学”

当前系统不是整本 `YTC`、也不是整本《专业投机原理》、也不是整本《交易圣经》的代码化身。

它只是从这些来源中，挑出了能冻结成正式合同、能跑正式验证、能进入主线 runner 的那一部分。

所以必须明确：

1. **书里说过，不等于系统已实现。**
2. **旧系统做过，不等于新系统已继承。**
3. **有 detector，不等于已经通过正式业务裁决。**

## 4. 第一层：市场本体层

这一层回答的是：

`市场现在处于什么背景、什么阶段、什么方向关系里？`

### 4.1 已正式定义并实现

#### 4.1.1 多重时间框架分析

状态：`已正式定义并实现`

当前正式映射不是 YTC 的日内三层，而是 A 股日线主线下的三层：

1. 月线：
   - `monthly_state_8`
2. 周线：
   - `weekly_flow_relation_to_monthly`
3. 日线：
   - `pas_trigger`

代码落点：

1. [contracts.py](G:\MarketLifespan-Quant\src\mlq\malf\contracts.py)
2. [bof_16cell.py](G:\MarketLifespan-Quant\src\mlq\alpha\pas\bof_16cell.py)
3. [trigger_16cell.py](G:\MarketLifespan-Quant\src\mlq\alpha\pas\trigger_16cell.py)

#### 4.1.2 顺流 / 逆流

状态：`已正式定义并实现`

当前正式口径：

1. `with_flow`
2. `against_flow`

兼容读数：

1. `MAINSTREAM`
2. `COUNTERTREND`

这已经不是研究语言，而是正式 readout 主轴。

#### 4.1.3 趋势

状态：`已正式定义并实现`

当前系统没有把“趋势”停留在口语层，而是冻结成：

1. `BULL_FORMING`
2. `BULL_PERSISTING`
3. `BULL_EXHAUSTING`
4. `BULL_REVERSING`
5. `BEAR_FORMING`
6. `BEAR_PERSISTING`
7. `BEAR_EXHAUSTING`
8. `BEAR_REVERSING`

也就是说：

**趋势在本系统里，已经不是泛泛的涨跌，而是正式分段状态机。**

### 4.2 已部分承担，但不是完整实现

#### 4.2.1 市场平均寿命评估

状态：`已部分承担`

我的判断：

1. `MALF` 的整体愿景，明显是在承担“市场平均寿命框架”。
2. 但当前系统正式冻结下来的，是市场背景分层与阶段标签。
3. 还没有一个单独、明确、稳定的“平均寿命估计器 / percentile 合同 / 正式报告口径”。

所以更准确的说法是：

**系统已经吸收了“市场寿命/阶段统计”思想，但还没有把“市场平均寿命评估”做成独立正式能力。**

#### 4.2.2 波段

状态：`已部分承担`

原因：

1. `MALF` 有中期状态与周线关系。
2. `trade` 有 swing high、protective low、days_without_new_high。
3. 多个 trigger 也都在局部窗口里依赖波段推进。

但目前没有独立、统一、可被全系统复用的正式“波段对象合同”。

#### 4.2.3 高点 / 低点

状态：`已部分承担`

原因：

1. 多个 detector 都直接使用 `adj_high / adj_low`。
2. trailing stop 也依赖 `last_swing_high` 与保护低点。

但目前这些高低点主要是：

1. trigger 内部语义
2. trade runtime 局部语义

还不是全系统统一的“高点/低点识别引擎”。

#### 4.2.4 支撑 / 阻力

状态：`已部分承担`

原因：

1. `BPB / PB / TST / CPB` 都有 support、support_band、support_hold 之类的局部定义。
2. 这说明系统在局部触发层已经用上了支撑/阻力语言。

但它们还只是：

1. 各 trigger 内部的局部结构判断
2. 不是统一抽象成一套系统级 support / resistance 合同

#### 4.2.5 强与弱、辨识强势和弱势

状态：`已部分承担`

原因：

1. `monthly_state_8` 与 `with_flow / against_flow` 已经在市场层表达了强弱关系。
2. 五个 trigger 也都有 `strength` 分值。
3. 历史 sidecar 里还有过强弱相关尝试。

但当前主线没有冻结一套通用的：

1. 强势股定义
2. 弱势股定义
3. 强弱辨识引擎

所以这部分只能算局部承担。

### 4.3 当前未正式实现

#### 4.3.1 相对强度

状态：`当前未正式实现`

这点必须说死：

1. 当前主线没有正式 `relative_strength` 合同。
2. 也没有正式 runner、正式 readout、正式结论。

所以现在不能把“强弱”误讲成“相对强度已经实现”。

## 5. 第二层：触发与结构层

这一层回答的是：

`什么时候值得出手？`

### 5.1 已正式定义并实现

#### 5.1.1 五个 PAS trigger 的代码定义

状态：`已正式定义并实现`

代码落点：

1. [detectors.py](G:\MarketLifespan-Quant\src\mlq\alpha\pas\detectors.py)

当前正式存在五个 detector：

1. `BOF`
2. `BPB`
3. `PB`
4. `TST`
5. `CPB`

这说明：

**五个 trigger 作为“代码层形态定义”，已经存在。**

但业务裁决要另看。

### 5.2 书义与系统实现的精确边界

#### 5.2.1 BOF

状态：`代码已实现 + 业务已正式验证`

当前系统里的 `BOF`，是一个做多方向的：

1. 支撑被向下假跌破
2. 当日收回
3. 收盘位置够强
4. 量能确认

这里我要替我们压一条专业边界：

**你把 `2B` 和 `BOF` 等同看，作为研究映射是成立的；但系统口径上，当前还不应该说“BOF = 2B”。**

原因：

1. `2B` 在《专业投机原理》里是更广义的趋势变化早期警报。
2. 当前代码里的 `BOF` 是一个更窄的、做多方向的支撑假跌破恢复模型。

所以正式说法应是：

**当前 `BOF` 与 `2B` 高度同源、可互相解释，但不是已经冻结为同一合同。**

#### 5.2.2 BPB

状态：`代码已实现 + 业务已正式验证`

书里给了很强的光环：

1. 经典
2. 最赚钱
3. 像勺子一样的标准机会

但系统正式结论已经很清楚：

1. `BPB` 代码有
2. 三年正式验证也跑完了
3. hardened 链路复跑后，仍然整体不行

所以正式口径只能是：

**`BPB` 的书义地位很高，但在我们当前系统和当前市场样本里，并没有被验证成主线可用 trigger。**

#### 5.2.3 PB

状态：`代码已实现 + 业务已正式验证`

当前正式结论：

1. `PB` 不是通吃 trigger
2. 它只在少数格子里保留

你补充的这条：

`上涨的第一个 PB 最安全`

我认为很值得认真对待，但现在必须归类为：

**研究假说，不是系统结论。**

因为当前代码和正式 readout 里还没有：

1. 第几个 PB 的计数合同
2. “第一 PB” 的独立验证卡

#### 5.2.4 TST

状态：`代码已实现，业务未正式验证`

你说它：

1. 本身反脆弱
2. 可能在区间里更好
3. 因为个股大多数时间都在区间

这些都很像研究上有道理的判断。  
但当前系统口径只能说：

1. `TST` detector 已经实现
2. 还没有独立正式验证

所以不能把“区间里更好”提前说成系统结论。

#### 5.2.5 CPB

状态：`代码已实现，但语义还有债务`

这是当前最需要保持谨慎的 trigger。

我的判断是：

1. 代码里确实有 `CPB` detector。
2. 但它目前更像“压缩整理后的突破”，而不完全等于书里复杂回调的原义。

所以当前正式口径只能是：

**`CPB` 有实现，但定义尚未完全对齐书义，更没有正式业务裁决。**

### 5.3 当前未正式实现

#### 5.3.1 “上涨的第一个 PB 最安全”

状态：`当前未正式实现`

结论很直接：

1. 这是一个强研究命题。
2. 当前系统还没有正式变量来承接它。
3. 它应进入未来独立验证卡，不应先写进主线。

#### 5.3.2 TST 在区间市场更优

状态：`当前未正式实现`

原因不是观点没道理，而是：

1. 还没有独立正式三年验证
2. 系统也还没冻结“区间市场”作为正式业务裁决对象

#### 5.3.3 CPB 反脆弱定义

状态：`当前未正式实现`

这部分要先定义，再验证，不能倒过来。

#### 6.2.1 期望公式

状态：`已部分承担`

1. 正式 readout 已经能产出胜率、平均净收益、交易数。

但当前系统没有把它冻结成独立合同字段或独立优化引擎。  
也就是说：

**期望可以被推导，但“期望公式”还不是系统一级对象。**

#### 6.2.2 资金管理 / 衡量风险以管理资金

状态：`已部分承担`

这是另一条需要谨慎表述的。

1. 风险单位
2. 固定名义仓
3. 整手控制
4. 部分退出
5. trailing

但当前系统还没有完整继承上一版更专门的资金管理实验体系，  
也没有把《交易圣经》里的整套资金管理哲学冻结成正式模块。

所以更准确的说法是：

**我们已经有主线风险控制合同，但还没有完整资金管理体系。**
